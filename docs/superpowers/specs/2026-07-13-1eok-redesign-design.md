# "1억 모으기" 개편 설계 (2026-07-13)

## 배경과 문제 정의

기존 앱은 "지출/수입을 매일 기록 → 실제 저축 속도 → 1억 도착 예상일" 구조였다.
브레인스토밍에서 확인한 구조적 문제:

1. **보상 부재**: 일상 지출 한 건은 도착 예상일을 0.4일 정도밖에 못 움직여, 기록의 즉시 보상이 없다.
2. **수동 기록의 지속 불가**: 기록을 며칠 빼먹으면 순자산 숫자가 현실과 어긋나고, 오차가 영구 누적된다. "하루 빼먹으면 전체가 거짓말이 되는" 설계는 유지되지 않는다.
3. **경쟁 환경**: 자동 수집(토스·뱅크샐러드)과 수동 기록으로 경쟁할 수 없다.

## 제품 방향 (확정)

**"한 달에 5분, 빼먹어도 안 망가지는 현실 거울"** + **"오늘 이길 수 있는 게임"**

### 3개 레이어

| 레이어 | 주기 | 역할 | 실패 허용 |
|---|---|---|---|
| **미션** | 매일 | "오늘 쓸 수 있는 돈" 재분배형 예산. 매일 여는 이유 | 초과해도 실패 없음, 내일 예산이 줄어들 뿐 |
| **기록** | 수시 | 지출/수입 입력 = 미션 현황 갱신. 고정비는 자동 기록 | 빈 날은 판정 없음, 스트릭 안 끊김 |
| **결산** | 월 1회 | 계좌별 잔고 입력 → 자산을 현실로 재동기화 → 월간 보고서 | 한 달을 빼먹어도 다음 결산이 진실을 복원 |

핵심 논리: 결산은 보상 장치가 아니라 **데이터 공급 + 재동기화 장치**다. 기록 오차가
누적되어도 결산이 매달 교정하므로, 기록은 부담 없는 옵션이 된다. 보상(보고서·그리드·
스트릭)은 월 1회 결산과 매일 미션을 지속시키기 위한 부속 장치다.

### 보상 원칙

- **정직한 보상만**: 돈이 아니라 행동을 칭찬한다. 지출 입력에 "목표에 다가갔습니다"류
  거짓 멘트 금지. "이번 달 14일째 기록" 같은 행위 연속성만 보상.
- 보상 주기와 실제 숫자가 움직이는 주기를 일치시킨다(일=미션, 월=결산·보고서).

## 계산 로직 (확정)

용어: `변동지출` = source가 "fixed"가 아닌 지출 거래.

- **필요 월저축** = (목표 금액 − 현재 자산) ÷ max(1, 목표일까지 남은 개월)
- **이번 달 변동지출 예산** = 월 수입 − 활성 고정지출 합 − 필요 월저축
  - 월 수입: settings.monthlyIncome (온보딩에서 입력, 활성 고정수입 합으로 프리필, 설정에서 수정 가능)
  - 결과가 0 이하면 미션 카드는 "이 목표일은 현재 수입으로 불가능" 상태를 표시하고 목표일 조정을 제안
- **오늘 예산 (재분배형)** = max(0, (이번 달 변동지출 예산 − 이번 달 1일~어제 변동지출) ÷ 오늘 포함 남은 일수)
- **오늘 잔여** = 오늘 예산 − 오늘 변동지출 (음수 가능, 음수면 초과 표시)
- **미션 판정**: 그날 변동지출 기록이 1건 이상 있고 `오늘 지출 ≤ 오늘 예산`이면 클리어.
  기록이 없는 날은 판정 없음(스트릭을 끊지도 늘리지도 않음). 과거 기록 추가/삭제 시
  판정은 결정적으로 재계산된다(모든 판정은 거래 이력에서 파생, 별도 저장 없음).
- **미션 스트릭** = 판정 있는 날들 중 최근 연속 클리어 수. 오늘도 현재까지의 지출로
  판정에 포함(초과 전까지는 클리어 중으로 표시)
- **현재 자산** = 최신 결산 총액 + 결산일 이후 기록된 (수입 − 지출)
  - 온보딩에서 계좌별 잔고를 입력받아 항상 결산이 최소 1건 존재
- **실제 월 저축(결산 델타)** = (이번 결산 총액 − 직전 결산 총액) ÷ 두 결산 사이 개월 수.
  결산을 빼먹은 달이 있어도 월평균으로 정규화되어 실패 허용 원칙과 일치
- **도착 예상일 속도** (우선순위):
  1. 최근 결산 델타 최대 3개의 평균 (델타 1개 이상이면 사용, 라벨 "최근 결산 기준")
  2. 델타가 없으면(결산 1건뿐) 필요 월저축을 가정값으로 사용하되 라벨을
     "목표 페이스 가정 · 첫 결산 후 실제 기준으로 전환"으로 명시 — 이때 도착일은
     목표일과 같으므로 정직하게 가정임을 드러낸다
- **도착 예상일** = 오늘 + ceil(남은 금액 ÷ 속도) 개월

## 화면 설계 (확정)

### 현실 탭 (첫 화면) — 위에서 아래로

1. **미션 카드** (히어로): "오늘 쓸 수 있는 돈" + 잔여 금액(대형 숫자) + 예산/사용 +
   진행바 + 클리어 스트릭
2. **− 지출 / + 수입** 버튼 (아웃라인 스타일)
3. **1억 카드**: 남은 금액, 현재 자산, 도착 예상일, 진행바. 계산 기준 라벨은 탭하면
   계산 기준 시트(기존 openSpeedSheet 개편)
4. **결산 카드**: "N월 결산 D-n" + **최근 12개월 롤링 그리드**(12칸) + 연속 결산 수.
   탭하면 결산 시트

그리드 셀 상태: 결산완료·델타≥필요저축(민트) / 결산완료·델타<필요저축(민트 40%) /
미결산(진회색) / 이번 달(민트 점선 테두리). 판정 기준은 해당 결산의
requiredSaving 스냅샷 — 이후 목표를 바꿔도 과거 셀은 변하지 않는다.

### 결산 시트 (월 1회 플로우)

계좌 목록별 잔고 입력(직전 결산 값 프리필, 합계 실시간 표시) → 저장 →
**월간 보고서 시트** 즉시 표시.

### 월간 보고서 (결산 즉시 생성, 흐름 탭에 보관)

- 자산 증감(델타), 계좌별 변화
- 실제 저축 vs 필요 월저축 (달성/미달, 분석가 톤 문구)
- 도착 예상일 변화 (앞당김/밀림, 개월 수)
- 저축률(델타 ÷ 월 수입), 연속 결산 수

### 흐름 탭

월간 보고서 목록(최신순) + 이번 달 카테고리 지출 + 결산 기반 저축 추세.
기존 변동자산 카드는 제거(결산으로 흡수).

### 기록 탭

기존 유지: 최근 거래 목록, 삭제. 입력 후 현실 변화 카드는 미션 잔여 중심으로 갱신.

### 설정 탭

목표(금액·목표일·월 수입) / **계좌 관리(추가·이름변경·비활성·순서)** /
고정 항목(기존) / 데이터(백업·복원·초기화).

### 온보딩 (신규 사용자) — 3스텝 + 피날레 위저드

`document.body.appendChild`로 붙는 모달(`openOnboardingWizard()`), render()와 분리되어
있어 setState가 일어나도 입력값이 초기화되지 않는다. 스텝 상태와 입력값은
클로저에 보관. `hasOnboarded`가 false이고 `.wizard-backdrop`이 없으면 부트스트랩과
render() 양쪽에서 자동으로 열려, 전체 초기화·v1 백업 복원 후에도 다시 뜬다.

1. **STEP 1 "현재 자산은?"** — 계좌 행(이름 + 잔고, 언더라인 입력) 2행으로 시작,
   "+ 계좌 추가"로 행 추가. 금액 입력마다 하단에 실시간 피드백
   "현재 자산 {readableMoney}". 이름 있는 계좌가 0개인 채로 "다음"을 누르면
   모달 내부 인라인 경고만 표시(앱 전역 render() 호출 없음). CTA "다음 →".
2. **STEP 2 "1억 달성 시점은?"** — 목표 금액 행(기본 1억, settings.targetAmount로
   프리필) + 시점 세그먼트 `[3년] [5년] [2030년] [직접] [저축액으로]`(선택 시
   민트+굵기, 밑줄 없음; 5개가 한 줄에 안 들어가 `flex-wrap: wrap; gap: 12px 20px;`).
   3년/5년은 `addMonths(today, 36/60)`, 2030년은 "2030-12-31", 직접은 date input.
   기본 선택은 `settings.targetDate`가 "2030-12-31"과 다르면 '직접'+그 값, 아니면 '5년'.
   실시간 피드백 "매달 {N}만원 페이스가 필요해요"(이미 목표 달성 시
   "지금 자산으로 이미 목표에 도달했어요"). CTA "다음 →".
   **역산 모드("저축액으로")**: "몇 년 안에 모을지 모르겠다"는 사용자를 위해 날짜
   대신 저축액으로 도착일을 역산. 날짜 input 대신 목표 금액 행과 같은 스타일의
   `월 저축액` 행 표시(placeholder "예: 1,200,000"). 실시간 피드백: 저축액 > 0이면
   `arrivalDate(남은 금액, 저축액, today)`로 "이 저축이면 {YYYY}년 {M}월에 1억
   달성해요", 비어 있으면 muted 톤 "월 저축액을 입력해 주세요"(남은 금액 0이면
   기존 "지금 자산으로 이미 목표에 도달했어요" 우선). "다음" 클릭 시 저축액 > 0을
   검증하고(아니면 진행 차단, 피드백이 곧 인라인 경고) `targetDate =
   localDateString(계산된 도착일)`로 확정 — 이후 STEP 3·피날레·제출은 파생된
   targetDate로 기존 로직을 그대로 탄다. 뒤로가기로 돌아와도 모드·저축액 입력은
   클로저에 보존.
3. **STEP 3 "한 달 현금흐름은?"** — 월 수입 행 + 고정지출 행(라벨에 '선택' 표기),
   피드백 없음. CTA는 민트 텍스트 "오늘의 미션 보기 →".
4. **피날레** — 보더 민트 계열(rgba(60,232,164,0.5)) 카드, 중앙 정렬. 라벨
   "오늘 쓸 수 있는 돈" → 그라데이션 큰 숫자(흰→민트, `{금액}원`, 오늘 예산 =
   `max(0, monthlyVariableBudget(월수입, 고정지출, 필요저축)) ÷ 이번 달 오늘 포함
   남은 일수`) → 문장 "이 페이스면 {YYYY}년 {M}월에 1억 달성합니다."
   (`arrivalDate(남은 금액, 필요저축, today)`가 없으면 문장 생략) → 민트 버튼
   "시작하기". `monthBudget ≤ 0`이면 숫자 대신 "이 목표일은 현재 수입으로
   어렵습니다. 시작 후 설정에서 목표일·월 수입을 조정해 주세요."를 표시하되
   시작하기는 그대로 동작.

각 스텝(1~3)에 진행 점 3개(민트 = 지나온/현재), STEP 2부터 우상단 "←"
뒤로가기(입력값 보존), 금액 input에서 Enter = 다음.

**확정 수치**: 카드 `width: min(88vw, 340px)` 중앙 배치, radius 8px, padding
`24px 18px 20px`(피날레 `32px 18px 24px`) · 진행 점 4px 원·간격 5px·아래여백 40px ·
질문 19px/800·아래여백 40px · 입력 행 라벨 13px muted/값 16px·700·tabular,
padding `13px 0 11px`, 밑줄 1px(포커스 시 민트), `min-width: 0`으로 flex 오버플로
방지 · 계좌 행은 이름:잔고 = `flex: 1.3 : flex: 1`(잔고 8자리가 잘리지 않게) ·
"+ 계좌 추가" 12px faint·위 12px ·
세그먼트(항목 5개) 15px·`flex-wrap: wrap`·`gap: 12px 20px`·위여백 22px · 피드백 13px 민트(비활성 안내는 muted)·위여백 15px · CTA
15px/700 오른쪽 정렬·위여백 38px · 피날레: 라벨 13px muted 중앙 → 16px → 숫자
40px/800 그라데이션(90deg, #fff 20% → #3ce8a4 85%) → 43px → 문장 13px
faint(line-height 1.7) → 46px → 시작하기 버튼(#3ce8a4 배경, #06281b 글자,
15px/800, padding 14px 0, radius 8px).

"시작하기" 제출은 기존 온보딩 병합 의미론을 그대로 유지: `...state` 기반 병합,
`fixedItems.length === 0`일 때만 월급/고정비 auto 항목 생성, 기존 transactions
보존, settlement에 requiredSaving(반올림)·monthlyIncome 스냅샷. 완료 즉시 미션
카드가 살아있는 첫 화면. **샘플 데이터는 완전 제거.**

## 기술 구조

- 의존성 없는 정적 웹앱 유지: index.html / styles.css / app.js / manifest.json
- **PWA 완성**: service worker(sw.js, 앱셸 캐시-우선, 배포 시 캐시 버전 갱신),
  iOS 메타태그(apple-mobile-web-app-capable, apple-touch-icon, status-bar black-translucent)
- **localStorage 새 키** `one_hundred_million_mobile_v2`:

```
{
  settings: { targetAmount, targetDate, monthlyIncome, startDate },
  accounts: [{ id, name, order, active }],
  settlements: [{ id, month "YYYY-MM", date "YYYY-MM-DD",
                  balances: [{ accountId, amount }], total,
                  requiredSaving, monthlyIncome }],  // 결산 시점 스냅샷 — 그리드 판정·보고서가 소급 변경되지 않게 고정
  transactions: [{ id, type, amount, category, title, date, source, fixedItemId? }],
  fixedItems: [{ id, type, name, amount, day, category, active, startMonth }],
  hasOnboarded: boolean
}
```
